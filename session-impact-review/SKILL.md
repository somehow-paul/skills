# Session Impact Review

只针对当前会话范围内的改动做审查，不把无关历史改动混入结论。

## 1. 使用时机

- 用户明确要求只审查“本次会话”“本次任务”“当前改动”
- 用户除了查代码问题，还要求判断相关功能是否会被影响
- 需要输出“影响范围 + 风险点 + 建议验证项”的组合结论

## 2. 核心边界

- 审查默认是只读行为，不主动修改代码
- 只覆盖当前会话范围内可识别的改动
- 如果无法准确隔离“当前会话”与“既有未提交改动”，必须明确说明降级依据
- 不把“未验证”说成“没问题”

## 3. 范围识别

优先按下面顺序确定审查范围：

1. 用户明确提供 `base commit`、`base branch`、PR 范围或补丁文件
2. 环境变量 `SESSION_REVIEW_BASE` 或 `REVIEW_BASE`
3. 使用脚本 `scripts/detect_scope.sh`
4. 如果仍无法隔离当前会话，只审查当前未提交改动，并明确写出局限性

### 推荐命令

```bash
bash /Users/paul/.codex/skills/session-impact-review/scripts/detect_scope.sh
```

或在已知基线时：

```bash
bash /Users/paul/.codex/skills/session-impact-review/scripts/detect_scope.sh <base-ref>
```

### 范围判定规则

- `SCOPE_MODE=base-ref`：审查 `base-ref...HEAD` 之间的提交改动，加上当前 staged 和 unstaged 改动
- `SCOPE_MODE=uncommitted-only`：只审查当前未提交改动
- 如果工作区里原本就有其他人留下的未提交改动，且无法区分是否属于本次会话，必须在结果中显式标注“范围存在混入风险”

## 4. 审查流程

### Step 1：锁定文件清单

先拿到本次会话范围内的文件列表：

- 为空：直接输出“无当前会话改动可审查”
- 非空：逐个文件进入 diff 与调用链分析

### Step 2：阅读差异而不是整仓扫射

必须优先阅读：

- `git diff -- <file>`
- 相关类型、接口、路由、配置、测试
- 变更附近的上下文代码

只有在定位调用链时，才扩展到 `rg` 检索调用方或被调用方。

### Step 3：做功能影响分析

每个改动文件都要回答两个问题：

1. 这个文件自身有没有引入缺陷、遗漏分支或契约破坏
2. 它会不会影响其他功能、页面、接口、任务、配置或数据流

### Step 4：按改动类型扩展检查

#### 接口 / 路由 / Controller

- 是否影响前端调用、权限、参数校验、返回结构
- 是否改变路径、字段名、枚举值、状态码、分页结构
- 是否漏改文档、SDK、调用方或测试

#### Service / Domain / Use Case

- 是否影响上游调用链
- 是否改变事务边界、异常语义、缓存、副作用、并发行为
- 是否存在遗漏的分支、空值、幂等、回滚问题

#### DTO / VO / Type / Enum

- 是否破坏序列化结构或跨端契约
- 是否新增字段但未同步默认值、转换器、校验器、展示层
- 是否出现魔法值回流、枚举映射不完整

#### Repository / SQL / Entity / Schema / Config

- 是否影响历史数据兼容性
- 是否引入 N+1、索引缺失、语义变更或启动失败风险
- 是否要求联动修改配置、脚本、环境变量

#### Frontend Component / Page / Store

- 是否影响路由跳转、props、emit、store 状态、API 参数
- 是否覆盖空态、加载态、错误态、权限态
- 是否引入 ID 精度问题、响应式丢失、条件渲染遗漏

#### 样式 / 布局 / 交互

- 是否影响响应式布局、滚动、层级、点击区域、主题或安全区
- 是否波及共享样式、公共组件或其他页面

### Step 5：验证建议或最小验证

如果仓库具备低成本验证能力，优先执行最小必要验证：

- 只跑与改动模块直接相关的 test / lint / typecheck / compile
- 不为追求“全绿”去跑明显无关的大而全命令
- 如果未执行验证，必须说明原因与残余风险

## 5. 影响范围输出要求

结果里必须单独列出“影响功能矩阵”，至少包含：

- 变更文件
- 直接影响的功能
- 间接影响的调用链或依赖项
- 当前判断：`已覆盖`、`需验证`、`有风险`
- 判断依据：代码位置、diff 事实、调用链证据

## 6. Finding 级别

- `P0`：安全漏洞、数据损坏、严重线上故障风险
- `P1`：明确的功能错误、契约破坏、关键场景遗漏
- `P2`：可维护性、性能、测试覆盖或边界处理不足
- `P3`：优化建议，不阻塞交付

满足以下任一条件，结论应为 `BLOCK`：

- 存在 `P0`
- 存在 `P1`
- 关键影响链路无法自洽，且高概率影响现有功能

如果没有阻塞问题，但关键链路尚未验证，结论应为 `NEEDS-VERIFY`。

只有在以下条件同时满足时，结论才可为 `PASS`：

- 未发现 `P0` / `P1`
- 影响功能矩阵没有高风险遗漏
- 已完成必要验证，或证据足够强能支撑当前结论

## 7. 输出模板

```text
SESSION IMPACT REVIEW

Scope Basis:
- mode:
- base:
- files:
- limitation:

Change Summary:
- ...

Impact Matrix:
- <file> -> 直接影响: ... | 间接影响: ... | 判断: 已覆盖/需验证/有风险 | 证据: ...

Findings:
- [P1] path:line - 问题描述
  - 影响功能:
  - 原因:
  - 建议修复:

Verification:
- 已执行:
- 未执行:

Final Verdict:
- PASS | NEEDS-VERIFY | BLOCK
```

## 8. 平台内联评审

如果当前运行环境支持内联评审指令，且发现明确问题：

- 每个问题输出一条内联评审注释
- 只对真正的问题发注释，不对普通说明发注释
- 行号必须尽量精确，优先标到最小范围

## 9. 审查纪律

- 不得把范围外文件纳入结论
- 不得只报代码风格而忽略真实功能风险
- 不得只看改动文件本身而不追踪调用链
- 不得在证据不足时下“无影响”结论
- 不得把修复建议写成大重构，优先最小修复面

## 10. 快速命令参考

```bash

# 查看范围内文件 diff

# 查调用链

```
