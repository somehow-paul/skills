# Commit With Changelog

## Overview

按模块组织提交，并把每个模块对应的用户价值写入 README 更新日志。执行时优先保证提交边界清晰、日志可读、非技术用户可理解。

## Workflow

### Step 1: 收集改动并划分模块

1. 读取当前变更文件列表与 diff。
2. 按业务模块分组改动，优先使用目录边界和业务语义双重判断。
3. 若同一文件涉及多个模块，只在语义可拆分时拆分提交；否则归入主模块并在日志中标注影响范围。

## Step 2: 生成分模块提交计划

1. 为每个模块定义单独提交目标。
2. 每个提交仅包含一个模块内的同类变化。
3. 若单模块改动过大，继续拆为多个提交（如新增能力、体验优化、问题修复）。

提交信息建议格式：
- `<模块名>: <用户价值导向标题>`

## Step 3: 执行提交

1. 逐模块暂存对应文件，避免混入跨模块改动。
2. 提交前再次核对 staged 文件是否与模块一致。
3. 完成一个模块提交后再处理下一模块。

## Step 4: 更新 README 的更新日志

1. 在 README 中定位或创建“更新日志”区域。
2. 以日期为分组写入本次发布内容。
3. 在日期下按模块写明功能点与体验变化。

日志写作规则：
- 只写用户可感知变化，不写实现过程。
- 使用业务语言，不使用技术术语堆砌。
- 每条功能点都要回答“用户现在能做什么”或“体验变好了什么”。

建议句式：
- “用户现在可以……”
- “页面/流程现在会……”
- “操作反馈变得更……”
- “本次更新让……更便捷/更稳定/更清晰”

禁止写入 README 更新日志的内容：
- 类名、方法名、文件路径、函数调用链。
- 数据库表结构、字段迁移、索引调整等实现细节。
- 具体框架改造、重构策略、底层缓存和接口技术细节。
- 面向开发者的调试信息。

## Step 5: 交付前检查

1. 每个模块至少对应一条用户价值日志。
2. 日志可被非技术角色直接理解。
3. 确认日志中不存在代码层术语或实现细节。
4. 提交顺序与日志模块顺序保持一致，便于追溯。

## Output Format

最终输出应包含：

1. 模块化提交结果列表（提交标题 + commit hash）。
2. README 更新日志的新增内容摘要（按模块列出）。
3. 若某模块仅有内部改动且无用户可感知变化，明确说明“本次未写入用户更新日志”。
